# 系统架构设计

## 1. 技术基线

- .NET 8、WPF、Windows x64。
- MVVM 组织 UI 状态；Core 层不引用 WPF。
- 绿色、自包含发布；所有转换逻辑离线运行。

## 2. 模块划分

| 模块 | 职责 |
|---|---|
| `XMacroBridge.Core` | 统一模型、接口、验证、嵌套展开和转换规则 |
| `Formats.Razer` | 雷云 XML 与 `.synapse4` 导入、雷云 XML 导出 |
| `Formats.Xmbc` | XMBC 配置/文本导入与宏文本导出 |
| `Application` | 用例编排、取消、进度、文件选择和导出事务 |
| `XMacroBridge.Presentation` | 与 WPF 无关的工作区 ViewModel、展示模型和 UI 状态编排 |
| `XMacroBridge.App` | WPF 视图、Windows 文件对话框、拖放和固定 Razer 暗色资源 |
| `Tests` | 单元、黄金样本、集成及 UI 测试 |

依赖方向为 App → Presentation → Application → Core；格式适配器依赖 Core 接口，Application 负责注册具体格式适配器。Core 和 Presentation 均不引用 WPF，Core 不依赖具体格式、文件系统或 UI。

## 3. 公共模型与接口

- `MacroDocument`：宏标识、名称、来源、事件、元数据和诊断。
- `MacroEvent`：键盘、鼠标、固定/随机延时、扫描码、XMBC 专属命令、宏引用和未知事件。
- `ConversionDiagnostic`：代码、严重级别、位置、消息和原始格式上下文。
- `TextEncodingDetector`：严格识别 UTF-8 与带 BOM 的 UTF-16 LE/BE，供格式探测与文本解码共用；不执行文件系统操作。
- `DiagnosticContext`：将格式适配器收到的来源路径收敛为脱敏文件名，禁止绝对路径进入文件级诊断上下文。
- `IMacroImporter`：探测并读取输入，返回文档集合和诊断。
- `IMacroExporter`：验证目标能力后生成输出。
- `INestedMacroResolver`：建立依赖图并输出线性宏。
- `IMacroValidator`：检查状态平衡、范围、资源限制和目标兼容性。

## 4. 数据流

输入始终先复制到只读流中解析。导入器生成统一模型；引用解析器构建依赖图并展开；验证器产生诊断；编辑器只操作统一模型；导出器将通过验证的模型写入临时文件，完成后再提交到用户选择的位置。

`Application` 层通过 `MacroFormatRegistry` 按内容与扩展名选择格式适配器。`MacroImportService` 安全展开文件/目录并执行批量限制，也接收内存中的 XMBC 文本并在 UTF-8 字节化后应用同一批次大小和格式路由；`SafeExportService` 负责原文件保护、同目录临时文件、磁盘刷新和原子替换。UI 不直接调用格式解析器或文件写入 API。

各格式导入器接收可选 `MacroLimits`，默认使用统一安全上限。事件解析循环在构造统一模型时即停止于 `MaximumEventsPerMacro`，不得依赖事后验证器才截断超长输入。

`WorkspaceViewModel` 维护宏列表、当前事件时间线、选中事件、宏重命名、宏文本替换、嵌套宏目标选择、受限撤销/重做历史、诊断、目标格式、进度和取消状态。宏重命名、文本替换、嵌套目标重新绑定及嵌套事件插入均以新的 `MacroDocument` 替换原条目并进入同一撤销历史；手动绑定写入目标宏 GUID、名称和当前工作区一基位置，GUID 仍是导出展开的首选事实来源。文本替换优先读取导入时保留在内存元数据中的 `xmbc.rawText`，否则通过 Application 层 XMBC 导出器生成文本，重新导入后保留原宏 ID、名称和来源路径并重建诊断。名称最长 128 字符；导出器始终读取当前文档名称，因此保存建议名和 XML `<Name>` 使用同一事实来源。`InputEventDisplayFormatter` 将统一键码、按钮和状态转换为受控事件类型文本。时间线插入、复制、删除、移动、搜索和撤销/重做在界面中可用；批量延时缩放与选中延时直接应用能力仍保留在 Presentation 层但不展示。App 层键盘捕获将 WPF `Key` 转为 Windows VK，并只对已确认的扩展键自动设置 `NewKeyIsExtended`，最终仍通过 ViewModel 的 0–255 校验和不可变编辑链路。

App 层直接加载唯一的 Razer 品牌暗色色板，不再包含运行时主题切换器、系统主题监听或测试主题覆盖参数。

App 启动时创建唯一主窗口，并将命令行中确实存在的文件或目录作为只读导入路径传给工作区。该入口与文件选择器、文件夹选择器、拖放和文本输入共用同一个 `MacroImportService` 管线，不允许形成绕过资源限制的第二套导入逻辑。

宏库由 Application 层 `MacroLibraryService` 管理：一级真实文件夹映射分组，`.txt/.xml` 使用相对路径标识，复制、文本保存、重命名、移动、回收和恢复均执行根目录范围检查。收藏、最近使用与回收站元数据位于宏库根目录的 `.xmacrobridge/`；应用级设置只在 `%LocalAppData%/XMacroBridge/settings.json` 保存当前根路径。`MacroLibraryViewModel` 负责双栏筛选、延迟格式预览、文件监视与操作状态；WPF 只提供对话框、导航和工作区协调。宏库打开条目时仍进入既有 `MacroImportService`，保存回库继续复用 `SafeExportService` 的验证与原子写入。

## 5. 错误处理

- 可恢复问题返回诊断，不使用异常控制正常流程。
- 单个宏条目或事件损坏时在最小边界隔离：保留同包其他宏及同宏前后有效事件，损坏事件转换为 `UnknownMacroEvent`；仅 XML/JSON 整体不良构时拒绝整份输入。
- 文件损坏、编码失败、资源超限等边界异常在应用层转换为中文错误。
- `Error` 阻止导出，`Warning` 要求用户可见，`Info` 仅说明规范化行为。
- 后台任务支持 `CancellationToken`，取消后释放文件句柄并删除临时输出。
