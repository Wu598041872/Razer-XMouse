# 开发工作流程

## 1. 开始任务

1. 阅读 `CLAUDE.md` 指定文档及最近日报。
2. 在 `docs/项目状态.md` 将一个明确任务移入“进行中”。
3. 确认验收条件、受影响格式和安全边界。
4. 若改变架构或兼容性契约，先新增 ADR。

## 2. 实施

- 一个变更只解决一个目标，避免格式研究、重构和 UI 改版混在同一提交。
- Core 层不得引用 WPF、文件对话框或具体格式实现。
- 新格式样本必须最小化和匿名化，并附预期结果。
- 业务错误使用 `ConversionDiagnostic`；未知事件不得用空 catch 忽略。

## 3. 验证

- 默认执行“最小充分验证”，验证范围与改动风险和影响面成比例。
- 文案、间距、颜色、对齐、边框、图标尺寸等小范围纯 UI 调整：只运行受影响 App 的基础编译，并完成一项针对性 WPF 冒烟或实机目视检查；不运行全量测试、自包含发布或完整设计 QA。
- 局部 UI 交互、绑定或 ViewModel 状态调整：运行受影响项目编译和直接相关的测试/冒烟，不扩展到无关格式与核心压力测试。
- 核心模型、格式导入导出、安全验证、文件写入或跨模块接口变化：运行相应项目测试；影响范围不确定时再逐级扩大验证。
- 格式变化运行双向黄金样本测试。
- 涉及文件输出时验证取消、重名和失败恢复。
- `tools/test.ps1 -Configuration Release` 仅用于跨模块高风险改动、构建/发布链路改动、阶段性里程碑或正式交付前验证，不得在每次微小 UI 修改后重复运行。
- 除非用户要求测试包或进入正式交付阶段，小改动后不重复生成自包含发布包、ZIP、哈希和完整截图对照。
- 同一轮开发需要交付可执行测试版本时，使用同一个测试构建目录和测试包反复覆盖更新；新修改继续在该构建中验证，不按时间戳或修改次数保留多个测试包。生成新测试包前先清理上一份测试构建。
- 只有正式版本发布，或用户明确要求同时保留多个方案用于对比时，才允许生成多个独立构建；正式版本不得覆盖历史正式发布物。
- 针对性验证失败、出现关联回归或无法判断影响面时，再升级到更高一级测试。
- 无法运行验证时记录阻塞原因与风险，不得宣称已通过。

## 4. 收尾

1. 更新相关规范、ADR 和 README。
2. 更新 `docs/项目状态.md`。
3. 运行日报 Session 模式，例如：

```powershell
./tools/devlog/Update-DevLog.ps1 -Mode Session `
  -Completed "完成某项工作" `
  -Tests "执行某项验证" `
  -Todo "下一步工作"
```

4. 检查 `git diff`，确保没有用户数据、秘密和构建产物。
5. 创建单一目的提交；后台任务不得自动提交或推送。

## 5. 分支与提交

- 稳定主分支为 `main`。
- 功能分支建议 `feature/<topic>`，修复分支建议 `fix/<topic>`。
- 提交标题采用简短祈使句，可使用中文；正文说明风险、测试和兼容性影响。
